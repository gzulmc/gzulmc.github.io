## Response:
适配器模式是一种结构型设计模式，它的核心思想是让两个不兼容的接口能够协同工作。就像现实生活中的电源适配器一样，当你的电器插头和插座不匹配时，你需要一个适配器来连接它们。

## 通俗易懂的例子

想象你有一个老式的VGA显示器，但你的新笔记本电脑只有HDMI接口。这时你需要一个VGA转HDMI的适配器：

```java
// 老式显示器只支持VGA接口
interface VGADisplay {
    void displayVGA(String content);
}

class OldMonitor implements VGADisplay {
    @Override
    public void displayVGA(String content) {
        System.out.println("通过VGA显示: " + content);
    }
}

// 新笔记本只有HDMI输出
interface HDMIOutput {
    void outputHDMI(String content);
}

class NewLaptop implements HDMIOutput {
    @Override
    public void outputHDMI(String content) {
        System.out.println("HDMI输出: " + content);
    }
}

// 适配器：将HDMI转换为VGA
class HDMIToVGAAdapter implements VGADisplay {
    private HDMIOutput hdmiOutput;
    
    public HDMIToVGAAdapter(HDMIOutput hdmiOutput) {
        this.hdmiOutput = hdmiOutput;
    }
    
    @Override
    public void displayVGA(String content) {
        // 在这里做转换工作
        System.out.println("适配器工作中：HDMI -> VGA转换");
        hdmiOutput.outputHDMI(content);
    }
}

// 使用示例
public class AdapterExample {
    public static void main(String[] args) {
        NewLaptop laptop = new NewLaptop();
        OldMonitor monitor = new OldMonitor();
        
        // 通过适配器连接
        VGADisplay adapter = new HDMIToVGAAdapter(laptop);
        adapter.displayVGA("Hello World!");
    }
}
```

## 在Java框架中的应用

### 1. Spring中的适配器模式

Spring框架大量使用了适配器模式，最典型的是**HandlerAdapter**：

```java
// Spring MVC中的HandlerAdapter
public interface HandlerAdapter {
    boolean supports(Object handler);
    ModelAndView handle(HttpServletRequest request, 
                       HttpServletResponse response, 
                       Object handler) throws Exception;
}

// 适配不同类型的Controller
@Component
public class SimpleControllerHandlerAdapter implements HandlerAdapter {
    @Override
    public boolean supports(Object handler) {
        return handler instanceof Controller;
    }
    
    @Override
    public ModelAndView handle(HttpServletRequest request, 
                              HttpServletResponse response, 
                              Object handler) throws Exception {
        return ((Controller) handler).handleRequest(request, response);
    }
}
```

Spring通过不同的HandlerAdapter来适配各种类型的处理器：
- `SimpleControllerHandlerAdapter` - 适配实现Controller接口的处理器
- `RequestMappingHandlerAdapter` - 适配@RequestMapping注解的处理器
- `HttpRequestHandlerAdapter` - 适配HttpRequestHandler接口的处理器

### 2. Spring Boot中的应用

Spring Boot的自动配置也大量使用适配器模式：

## 适配器模式的优点

1. **解耦性** - 让不兼容的类能够合作，无需修改原有代码
2. **复用性** - 可以复用现有的类，即使它们的接口不匹配
3. **灵活性** - 可以轻松切换不同的实现
4. **单一职责** - 适配器只负责接口转换，不影响业务逻辑

适配器模式在实际开发中非常常见，特别是在需要集成第三方系统、支持多种实现方式或者处理遗留系统时，它能够很好地解决接口不兼容的问题。

## Response:
很好的问题！从表面看确实都是"包装一下接口然后返回输出"，但它们的**目的和关注点**完全不同。让我用具体例子来说明区别：

## 核心区别

**适配器模式**：解决**接口不兼容**问题，让两个本来无法协作的类能够一起工作
**代理模式**：在**不改变原有接口**的前提下，控制对象的访问或增强功能

## 直观对比例子

### 适配器模式 - 接口转换
```java
// 老系统的打印机接口
interface OldPrinter {
    void printOldFormat(String text);
}

// 新系统的打印接口  
interface ModernPrinter {
    void print(Document doc);
}

// 适配器：让老打印机能在新系统中使用
class PrinterAdapter implements ModernPrinter {
    private OldPrinter oldPrinter;
    
    public PrinterAdapter(OldPrinter oldPrinter) {
        this.oldPrinter = oldPrinter;
    }
    
    @Override
    public void print(Document doc) {
        // 关键：接口转换！将Document转为String
        String text = doc.getText();
        oldPrinter.printOldFormat(text);  // 调用老接口
    }
}
```

### 代理模式 - 访问控制
```java
// 同样的打印接口
interface ModernPrinter {
    void print(Document doc);
}

// 真实的打印机
class RealPrinter implements ModernPrinter {
    @Override
    public void print(Document doc) {
        System.out.println("正在打印: " + doc.getText());
    }
}

// 代理：增加权限控制
class PrinterProxy implements ModernPrinter {
    private RealPrinter realPrinter;
    private String currentUser;
    
    public PrinterProxy(String user) {
        this.currentUser = user;
        this.realPrinter = new RealPrinter();
    }
    
    @Override
    public void print(Document doc) {
        // 关键：增强功能！添加权限检查
        if (!hasPermission(currentUser)) {
            throw new SecurityException("无权限打印");
        }
        
        System.out.println("代理：记录打印日志");
        realPrinter.print(doc);  // 调用相同接口
        System.out.println("代理：打印完成");
    }
}
```

## 关键区别总结

| 方面 | 适配器模式 | 代理模式 |
|------|------------|----------|
| **主要目的** | 接口转换，让不兼容的接口能协作 | 控制访问，增强功能 |
| **接口变化** | 实现不同的接口（接口转换） | 实现相同的接口 |
| **关注点** | 解决兼容性问题 | 增加额外功能（权限、缓存、日志等） |
| **使用场景** | 集成第三方系统、老系统改造 | 权限控制、延迟加载、缓存 |

## 总结

- **适配器**：像"转换器"，把A接口转成B接口，解决"不匹配"问题

简单记忆：
- 适配器 = 接口转换器（让不同的东西能一起工作）

## MyBatis代理的详细机制

### 1. 动态代理创建过程
```java
// MyBatis内部的代理工厂（简化版）
public class MapperProxyFactory<T> {
    private final Class<T> mapperInterface;
    
    public T newInstance(SqlSession sqlSession) {
        // 创建代理处理器
        MapperProxy<T> mapperProxy = new MapperProxy<>(sqlSession, mapperInterface);
        
        // 使用JDK动态代理创建接口实现
        return (T) Proxy.newProxyInstance(
            mapperInterface.getClassLoader(),
            new Class[]{mapperInterface},
            mapperProxy
        );
    }
}

// 代理处理器
public class MapperProxy<T> implements InvocationHandler {
    private final SqlSession sqlSession;
    private final Class<T> mapperInterface;
    
    @Override
    public Object invoke(Object proxy, Method method, Object[] args) {
        // 关键：将接口方法调用转换为SQL执行
        String statement = mapperInterface.getName() + "." + method.getName();
        
        if (method.getAnnotation(Select.class) != null) {
            return sqlSession.selectOne(statement, args[0]);
        } else if (method.getAnnotation(Insert.class) != null) {
            return sqlSession.insert(statement, args[0]);
        }
        // ... 其他SQL类型处理
    }
}
```

## 为什么需要代理？

### 问题：接口没有实现类

### 解决方案：动态代理
MyBatis通过代理模式，**自动为每个Mapper接口生成实现类**，省去了手动编写实现类的麻烦。

### 2. MyBatis代理的具体实现

```java
public class MapperProxy<T> implements InvocationHandler {
    private final SqlSession sqlSession;
    private final Class<T> mapperInterface;
    private final Map<Method, MapperMethod> methodCache;
    
    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        // 1. 构建Statement ID：namespace + method name
        String statementId = mapperInterface.getName() + "." + method.getName();
        // 结果：com.example.mapper.UserMapper.findById
        
        // 2. 获取方法对应的SQL配置信息
        MapperMethod mapperMethod = getMapperMethod(method);
        
        // 3. 根据SQL类型执行相应的SqlSession方法
        return mapperMethod.execute(sqlSession, args);
    }
    
    private MapperMethod getMapperMethod(Method method) {
        MapperMethod mapperMethod = methodCache.get(method);
        if (mapperMethod == null) {
            mapperMethod = new MapperMethod(mapperInterface, method, sqlSession.getConfiguration());
            methodCache.put(method, mapperMethod);
        }
        return mapperMethod;
    }
}
```

### 3. MapperMethod：核心转换逻辑

```java
public class MapperMethod {
    private final SqlCommand command;  // SQL命令信息
    private final MethodSignature signature;  // 方法签名信息
    
    public Object execute(SqlSession sqlSession, Object[] args) {
        Object result;
        
        // 根据SQL类型选择对应的SqlSession方法
        switch (command.getType()) {
            case INSERT: {
                Object param = signature.convertArgsToSqlCommandParam(args);
                result = sqlSession.insert(command.getName(), param);
                break;
            }
            case UPDATE: {
                Object param = signature.convertArgsToSqlCommandParam(args);
                result = sqlSession.update(command.getName(), param);
                break;
            }
            case DELETE: {
                Object param = signature.convertArgsToSqlCommandParam(args);
                result = sqlSession.delete(command.getName(), param);
                break;
            }
            case SELECT:
                if (signature.returnsVoid() && signature.hasResultHandler()) {
                    // void方法 + ResultHandler
                    executeWithResultHandler(sqlSession, args);
                    result = null;
                } else if (signature.returnsMany()) {
                    // 返回List/Array
                    result = executeForMany(sqlSession, args);
                } else if (signature.returnsMap()) {
                    // 返回Map
                    result = executeForMap(sqlSession, args);
                } else {
                    // 返回单个对象
                    Object param = signature.convertArgsToSqlCommandParam(args);
                    result = sqlSession.selectOne(command.getName(), param);
                }
                break;
            default:
                throw new BindingException("Unknown execution method for: " + command.getName());
        }
        return result;
    }
}
```

## 关键点总结

1. **Statement ID规则**：`namespace + "." + methodName`
2. **方法签名分析**：返回类型决定调用哪个SqlSession方法
3. **参数处理**：方法参数转换为SQL参数
4. **SQL类型判断**：从XML配置或注解中获取SQL类型
5. **结果映射**：JDBC结果集映射为Java对象

这就是为什么你的Mapper接口方法名必须和XML中的`id`一致，namespace必须是接口的全限定名的原因 —— MyBatis就是通过这种映射关系来找到对应的SQL配置的！
